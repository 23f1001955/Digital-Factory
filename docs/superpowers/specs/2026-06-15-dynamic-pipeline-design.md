# Dynamic Pipeline System

**Date:** 2026-06-15
**Status:** Draft
**Product:** Digital Product Factory

## Problem

Currently, 7 product types have fully hardcoded component lists in `schemas/*.json`. Every run of a product type produces the exact same components regardless of niche, market research, or user needs. This is rigid — a real estate OS and a SaaS OS get identical outputs.

## Solution Overview

Replace hardcoded component lists with a **hybrid approach**: a minimal set of core components defined in the schema, plus a fully dynamic `pipeline_plan` generated by the market_agent at runtime based on real market research data.

## Architecture

### 1. Minimal Schemas

Each schema JSON file will contain only core/infrastructure components:

```json
{
  "product_type": "operating_system",
  "display_name": "Operating System",
  "notion_sync": true,
  "components": [
    {"id": "market_research", "agent": "market_agent", "output": "data/market_research.json", "depends_on": []}
  ]
}
```

### 2. Core Components by Category

These are fixed per schema and ship with every run of that product type:

| Component | Applicable To | Condition |
|-----------|---------------|-----------|
| `market_research` | ALL | Always first |
| `images` | ALL | Always |
| `package` | ALL | Always last (wraps everything) |
| `notion_schema` | OS, Workflow Kit, Course | `notion_sync: true` |
| `notion_tree` | OS, Workflow Kit, Course | `notion_sync: true` |
| `gumroad_research` | ALL | Wizard: `gumroad_enabled` |
| `gumroad_publish` | ALL | Wizard: `gumroad_enabled` |
| `landing_page` | ALL | Wizard: `landing_enabled` |
| `social_promotion` | ALL | Wizard: `social_enabled` |

Images depends on market_research. Notion depends on notion_schema. All wizard-gated components keep their existing gating logic.

### 3. market_agent Generates pipeline_plan

market_agent's output (`market_research.json`) gains a new `pipeline_plan` field. **The LLM generates different components per niche** — this is not a fixed template:

#### Real Estate Agency
```json
{
  "niche": "real estate agency",
  "revenue_model": "commission",
  "team_size": 5,
  "pipeline_plan": {
    "components": [
      {"id": "property_crm", "agent": "csv_export_agent", "output": "data/property_crm.csv", "depends_on": ["market_research"]},
      {"id": "lead_tracker", "agent": "csv_export_agent", "output": "data/lead_tracker.csv", "depends_on": ["property_crm"]},
      {"id": "email_templates", "agent": "content_agent", "output": "content/email_templates.md", "depends_on": ["market_research"]}
    ]
  }
}
```

#### SaaS Startup
```json
{
  "niche": "SaaS analytics platform",
  "revenue_model": "subscription",
  "team_size": 12,
  "pipeline_plan": {
    "components": [
      {"id": "churn_analysis", "agent": "csv_export_agent", "output": "data/churn_analysis.csv", "depends_on": ["market_research"]},
      {"id": "onboarding_checklist", "agent": "content_agent", "output": "content/onboarding_checklist.md", "depends_on": ["churn_analysis"]}
    ]
  }
}
```

#### Coaching Business
```json
{
  "niche": "online coaching",
  "revenue_model": "course",
  "team_size": 2,
  "pipeline_plan": {
    "components": [
      {"id": "student_tracker", "agent": "csv_export_agent", "output": "data/student_tracker.csv", "depends_on": ["market_research"]},
      {"id": "curriculum_plan", "agent": "content_agent", "output": "content/curriculum_plan.md", "depends_on": ["market_research"]},
      {"id": "session_scheduler", "agent": "content_agent", "output": "content/session_scheduler.md", "depends_on": ["market_research"]}
    ]
  }
}
```

The LLM composes these from scratch based on niche research — no two niches get identical component lists unless the research justifies it.

Each component must specify: `id`, `agent`, `output`, `depends_on`. Optionally: `template` (for render_agent), `format`.

No `prompt` field needed — content_agent already resolves prompts by `component.id + ".j2"` (e.g., `id: "email_templates"` → loads `prompts/email_templates.j2`).

### 4. Agent Catalog in Prompt

market_agent prompt includes a catalog of available agents so the LLM can compose valid pipelines:

| Agent | Purpose | Requires Template | Output Format |
|-------|---------|-------------------|---------------|
| research_agent | Database + source compilation | No | JSON / Markdown |
| content_agent | Markdown content generation | No (prompt by id) | `.md` |
| image_agent | Image generation with SVG fallback | No | `.jpg` / `.png` / `.svg` |
| render_agent | HTML → PDF rendering | Yes (`template` field) | `.pdf` |
| csv_export_agent | JSON → CSV export | No | `.csv` |
| diagram_agent | Mermaid diagrams | No | `.mmd` |
| catalog_agent | Product catalog generation | No | `.json` |
| stitch_agent | Stitch AI design integration | No | Various |

Core agents (market, packaging, gumroad, landing, notion, notion_schema, social) are NOT in this catalog — they are managed by the orchestrator as fixed components.

### 5. Orchestrator Changes

**Flow:**

```
1. Load schema from schemas/{product_type}.json
2. Extract fixed components from schema
3. Run market_agent (always first)
4. Load market_research.json
5. Extract pipeline_plan from market_research.json
6. Validate pipeline_plan:
   a. Every agent in AGENT_REGISTRY
   b. Every component depends on market_research or another valid component
   c. No circular dependencies
   d. No duplicate component IDs
7. Merge: fixed_components + pipeline_plan.components
8. Apply wizard gates: skip gumroad/landing/social if disabled
9. Topo-sort full list
10. Execute normally (existing error isolation + resumability)
```

**Validation rules:**

- market_research is always first and must have no dependencies
- pipeline_plan components cannot depend on wizard-gated components
- All agent names must match AGENT_REGISTRY keys exactly
- Component IDs must be unique across merged list (no overlapping with core component IDs)
- Forbidden IDs (reserved for core components): `market_research`, `images`, `package`, `notion_schema`, `notion_tree`, `gumroad_research`, `gumroad_publish`, `landing_page`, `social_promotion`

### 6. Error Handling

Existing error isolation is unchanged:
- If market_agent fails → whole run aborts (no pipeline_plan = no dynamic components)
- If a dynamic component fails → only its dependents skip, rest continues
- Resumability: completed dynamic components in `job_state.json` are skipped on re-run

### 7. Prompt Template for market_agent

The `prompts/market_research.j2` template will be updated with a new section at the end that instructs the LLM to generate a `pipeline_plan` based on the niche and research data. It includes the agent catalog and constraints.

The prompt must include:
- The agent catalog table
- Constraint: all dynamic components must depend on `market_research` or another dynamic component with a valid dependency chain
- Constraint: component IDs must be unique and descriptive
- Constraint: output paths must follow conventions (`data/` for data files, `content/` for markdown)
- Direction: generate different components for different niches (real estate needs different things than SaaS)
- Output format must be valid JSON in the `pipeline_plan` key

## Implementation Order

1. Update `prompts/market_research.j2` — add pipeline_plan instruction + agent catalog
2. Update `orchestrator/orchestrator.py` — add pipeline_plan merge logic after market_agent runs
3. Update `schemas/*.json` — strip to core components only
4. Update tests
5. Create any initial prompt templates for common dynamic components

## Out of Scope

- pipeline_plan does NOT support nested/recursive pipelines (flat list only)
- No UI for pipeline plan editing (users trust the LLM)
- Existing wizard questions unchanged
- No automatic prompt template creation — prompts must exist before a component references them
